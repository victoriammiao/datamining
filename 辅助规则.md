# JC3503 作业协作 — 辅助规则（行动指南）

本文档汇总执行 AI 在本项目中必须遵守的约定，并与课程作业架构指南对齐。具体章节任务以用户在对话中的直接指示为准；勿自行套用文档中可能提及的「示例章节」指令。

---

## 一、协作与流程

1. **瀑布式开发**：按模块推进；每完成一个模块需由用户验收并就下一步讨论后再继续，不得在未确认的情况下串联多个未约定的模块。
2. **任务来源**：具体要做哪一节、改哪些内容，以用户在对话中的明确说明为准；不因其他文档中的「执行指令」示例而擅自执行未指派任务。
3. **文件输出**：未经用户说明，不自行新增说明类 Markdown 或其他说明文件；用户明确要求的文件（如本规则、记忆文件）除外。
4. **「注」的用法**：用户在对话中以「注」引出或紧随其后的说明（如「注，只需要简要描述……」）**仅作协作约定**，不得写入 Notebook 正文、作业交付表述或对外可见的分析段落，除非用户明确要求纳入正文。

## 二、核心叙事与分析闭环（The Loop）

每一段分析须遵守 **「不污染语序」**：在对应代码运行并产出结果之前，不得引用或讨论该代码将产生的具体数值。

循环结构：

1. **策略规划 (Strategic Planning)**  
   - **Motivation**：本步骤在数据挖掘路径中的必要性。  
   - **Expectation（可选）**：当结果较不确定时，可给出基于常识或已有发现的简要预期；若步骤结果具有必然性（如「数据清晰」类），可省略。

2. **代码实施 (Implementation)**：整洁、模块化、注释清晰的 Python；代码与注释 **全英文**，代码单元格内 **禁止中文**。

3. **结果分析 (Observation & Analysis)**  
   - **Results**：客观描述图表或表格呈现的内容。  
   - **Inference**：批判性思考 — 市场规律、机制或数据质量问题。  
   - **Next Step**：由当前发现自然引出下一分析动作。

以上四点可用**连贯论文段落**表达，不必在正文反复使用「Motivation / Expectation」等标签式小标题；但**逻辑顺序不变**：代码未跑完前不写死该代码才会出现的具体数值。

7. **一图一循环**：每张图单独完成「方案说明 → 代码（仅该图）→ 运行得到输出 → 观察 → 分析 →（必要时）下一步方案」，再进入下一张图；禁止在同一代码单元内连续绘制多幅无依赖关系的图后再统一写观察。

8. **列表与小标题节制**：`Steps`、`Observation` 等小节标题下，若要点 **不足三条**，用连贯段落书写，不用「小标题 + 无序列表」；一图一循环后，单步观察通常一至两点即可写清。

## 二点五、经验驱动：输出先行（增补，重要）

1. **由输出驱动，而非由提纲反推**：用户给出的步骤是**思维顺序**参考，写作时应像真实探索——先提出可检验的问题 → 运行代码得到表/数 → 再基于**已打印的具体计数、比例、分位数、日期范围**写解读与「为何下一步要看 X」。禁止写成「已知存在 1–N 个问题」再一次性堆代码、最后用笼统形容词总结。

2. **结论要有锚点**：凡写「很多 / 极大 / 严重右偏」等，必须能指回上文输出中的**具体数字**（如缺失行数与占比、零值行数与占比、均值与中位数、偏度系数、规则命中行数等）。

3. **执行与续写**：在可行时，应在终端或脚本对**同源数据**运行与 Notebook 等价的片段以核对输出后再写结论；若当前环境无法执行，应**中断**并请用户运行相关单元后粘贴输出，再继续——不得编造与未运行结果不一致的数字。

4. **【test log】**：允许为核对而打印中间变量和结果数值；统一用 **`【test log】`** 标记（注释或 `print` 前缀均可），便于提交前批量清理。如有利于你的分析，这个输出可以足够详细，甚至比原本可视化的输出还详细，有助于深度理解。

5. **叙事节制**：若某检查未发现异常（如日期解析无失败），不必渲染成亮点；笔墨留给确有信息量的偏离或矛盾。

6. **范围克制**：「后续研究须限定时间窗」等表述，仅在研究问题与时间设计已落地时展开；在分区审计未完成前，避免用远期方法论占位冲淡本节应有的实证结论。

## 三、数据安全与技术栈

1. **原始数据保护**：使用 `df_raw = pd.read_csv('games.csv')` 加载，并立即 `df = df_raw.copy()`，在副本上操作。  
2. **检查点**：大规模清洗或特征工程后，建议显式备份（如 `df_cleaned`），避免后续步骤污染。  
3. **技术栈**：Pandas；Seaborn / Matplotlib；SciPy（统计检验）；Scikit-learn（预处理与建模）。

---

## 四、可视化与表格（Publication-Ready）

1. **图表**  
   - 坐标轴须有标签与单位（如 Price ($)，Playtime (Minutes)）。  
   - 全局 `sns.set_theme(style="whitegrid")`；**横轴刻度仅在标签过长或过多时**顺时针旋转约 **45°**（用 `rotate_x_labels_if_needed` 等条件判断）；短标签、数值刻度、少量类目**禁止**为「统一风格」而旋转。  
   - 按变量类型选图：分类 — 条形图；连续 — 直方图 / KDE；关系 — 散点 / 热图等。  

2. **表格**：倾向 APA 风格三线表思路，减少冗余竖线；尽量用 `df.style` 美化输出。

3. **可视化密度**：积极用图支撑复杂分析；避免重复表达同一信息（换呈现方式须有不同目的）；禁止「一张图能说清却拆成多张」。

## 五、语言与双语（Markdown 正文）

- **第一段**：英文（正式可提交表述）。  
- **第二段**：中文简体（对照、便于核对逻辑）。  
用户可在定稿后删除中文块以提交纯英文作业。
注意，英文块和中文块属于不同的markdown块。

- **行文**：见 **§二点五**；避免套话式模板，但保持先英后中与「代码前不写死具体数值」。

---

## 六、深度与质量导向（高分方向）

1. 从仅报告准确率转向 **推断与解释**：特征重要性、系数符号与现实含义。  
2. **数据审计**：除缺失值外，关注逻辑矛盾（如价格为 0 而折扣非 0、峰值在线极高但均游玩时长为 0 等），并说明处理理由。  
3. **关键特征**：如 `Estimated owners` 等区间字符串，须在预处理中转为可用数值（如中值）以支持回归、相关性热图等。

## 七、记忆同步（`辅助记忆.md`）

在根目录 `辅助记忆.md` 中，用 **序号 + 一两句简要描述 + 时间** 记录完成事项，供不同对话间同步上下文。

- **时间格式**：本地时间，**精确到分钟**：`YYYY-MM-DD HH:mm`（示例：`2026-05-05 03:06`）。  
- **写入时机**：完成一项可交付动作或规则/记忆变更后追加一条；若需统一校正旧条目，在记忆中用新条目说明即可，避免静默改写历史描述。
