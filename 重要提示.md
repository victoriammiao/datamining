### 一、 核心定位：模型是“透视镜”，而非“产品”
在阿伯丁大学的课程体系中，数据挖掘的目标不是为了获得一个完美的预测结果，而是通过建模过程实现**知识发现（Knowledge Discovery）**。

*   **原则**：发现（Insight） > 精度（Accuracy）。
*   **目标**：利用算法量化变量关系，通过模型参数（如权重、重要性、系数）来解释游戏市场的底层逻辑。
*   **禁忌**：禁止只跑代码而不进行 Markdown 形式的“批判性解释”。

---

### 二、 机器学习任务的深度定位
每一种方法在你的 Notebook 中都应承载特定的分析任务：

| 阶段 | 方法/算法 | 挖掘目的与 Insight |
| :--- | :--- | :--- |
| **第一部分：推断基础** | **NHST (显著性检验)** | 验证 EDA 发现的规律是“真实存在”还是“随机巧合”。 |
| **第二部分：监督学习** | **决策树 (Decision Tree)** | 识别哪些特征（如价格、标签）是决定游戏成败的**关键节点**。 |
| **第二部分：监督学习** | **朴素贝叶斯 / SVM** | 探索特征空间的**可分性**，研究非线性边界下的分类逻辑。 |
| **第三部分：无监督学习** | **K-Means 聚类 / PCA** | 发现市场中潜藏的**自然群体**（例如：识别“高价高质”与“低价走量”的游戏群组）。 |
| **第三部分：无监督学习** | **关联规则 (Apriori)** | 挖掘游戏标签、功能之间的**共现关系**（如：哪些标签经常被打包出现）。 |
| **第四部分：特殊挖掘** | **时间序列 / 文本挖掘** | 分析游戏热度的**季节性趋势**或从用户评价中提取**情感特征**。 |

### 三、 核心冲突与批判性思考点 (Critical Thinking Points)
在 Level 0 要求下，你需要主动在 Notebook 中讨论以下潜在的方法论矛盾：

1.  **视觉 VS 统计**：可视化中两组数据看起来不同，但 T-检验显示不显著。
    *   *反思点*：是样本量（N）太小，还是方差（Variance）太大导致了结论的不确定性？
2.  **预测精度 VS 可解释性**：黑盒模型（如 SVM）预测准，但白盒模型（决策树）更容易理解。
    *   *反思点*：在分析 `games.csv` 时，我们是为了预测一个数值，还是为了给开发者提供可操作的建议？
3.  **异常值的双重性**：
    *   *反思点*：Outliers（如爆款游戏）在统计上可能干扰模型，但在商业挖掘中，它们可能蕴含了最成功的商业模式。

### 四、 推荐的分析工作流 (Workflow)

1.  **数据审计 (Data Audit)**：整体到局部观察，发现缺失值与异常分布。
2.  **预处理 (Preprocessing)**：清洗数据，并记录每一个处理决定背后的逻辑。
3.  **探索性分析 (EDA)**：通过可视化提出初步猜想。
4.  **统计验证 (Testing)**：使用 NHST 验证猜想的显著性。
5.  **建模挖掘 (Modeling)**：应用监督/无监督算法提取深层模式。
6.  **批判性总结 (Reflection)**：评估模型局限性，给出最终基于数据的洞察。

*   **Why**：为什么选这个算法？
*   **So What**：结果告诉了我们什么？
*   **Next**：基于此发现，下一步该做什么？
